# Tori 核心实体与操作逻辑设计

本文档定义 Tori Dashboard 中核心业务实体的产品流转逻辑，确保前端交互不仅是 CRUD 表单，而是反映真实的业务和数据流向。

## 1. Claim Session (短效验证码)

- **定位**：对用户**不可见**的内部机制。
- **业务逻辑**：它只是一个用于跨端（IM -> Web）确认身份的短效验证码行为。
- **界面呈现**：不应该提供列表视图让普通用户去管理或查看状态。界面上仅提供一个“输入验证码以完成绑定 (Redeem Token)”的极简入口。

## 2. Connection (外部账号连接)

- **定位**：用户在 Web 侧管理外部平台数据源（如 Steam）的凭证。
- **创建工作流**：必须是一个向导流程，而非数据库字段裸露。
  1. **选择平台 (Platform)**：用户选择要接入哪个外部平台（例如：Steam）。
  2. **选择代理 (Token-Proxy，可选)**：用户选择是否通过系统注册的 `token-proxy` 进行托管接入，或选择直连。
  3. **录入与授权**：
     - 如果选择了 Proxy：跳转至 `token-proxy` 提供的 OAuth/扫码页面，通过 callback 机制完成信息录入。
     - 如果选择了直连或只需公开 ID：提供简单的输入框让用户手动填入必要信息（如 Steam64 ID）。
- **管理动作**：列表支持增删改查。针对已有连接，详情页需支持诊断与手动同步（如 Fetch Profile）。

## 3. Proxy Instance (代理实例)

- **定位**：管理员用于统一接管外部平台 OAuth 流程的基础设施。
- **录入工作流**：极其简单的运维配置。
  - 用户手动填入 Proxy 的 URL（必要时提供 Token）。
  - 点击“连接”，系统立即请求该 URL，拉取并展示该 Proxy 支持哪些平台（Providers），并展示健康状态。

## 4. Subscription (事件订阅)

- **定位**：用户定义系统“在什么情况下，把什么数据，发给谁”。
- **界面呈现（数据流向可视化）**：不仅需要基础的表格，更要在列表或详情中明确呈现**数据链路**：
  - **触发源 (Trigger)**：这个通知从哪里触发？（可能是关联了某个后台的 TaskDef，也可能是监听了某个特定的 Connection）。
  - **目的地 (Destination)**：通知最终发往哪个物理频道（Channel）？
  - **投递路径 (Path)**：通知通过哪个具体的 Bot Instance，利用哪条 Channel Binding 送达到目标的？
- **价值**：排障时，用户能一眼看出是数据源挂了，还是发消息的 Bot 死了。

## 5. TaskDef & TaskRun (后台任务)

- **定位**：系统的底层履约引擎。
- **业务逻辑**：普通用户**不需要也不能**凭空“创建”一个 Task。
- **触发机制**：用户配置了特定的 Command，或者创建了某个需要定时轮询的 Subscription（比如定期检查 Steam 库）时，系统在底层自动生成并调度相应的 TaskDef。Dashboard 的 Task 界面仅用于管理员查看这些预设或自动生成的任务及其执行情况（TaskRun）。
